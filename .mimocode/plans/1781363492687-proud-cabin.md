# PasarGuard Full-Stack Code Audit Plan

## Audit Summary

Comprehensive audit of the PasarGuard codebase (Python/FastAPI backend + React/TypeScript frontend) covering security vulnerabilities, bugs, performance bottlenecks, and resource leaks. All fixes are backward-compatible and PR-ready.

## Finding 1: CRITICAL — Python 2-Style Exception Syntax (12 instances)

**Severity: Critical**
**Impact:** In Python 3, `except X, Y:` catches exception X and *binds it to variable Y* — it does NOT catch X or Y. The second exception type is never caught, causing silent bug masking and potential crashes.

### All locations to fix:

| # | File | Line | Current | Fix |
|---|------|------|---------|-----|
| 1 | `app/utils/jwt.py` | 41 | `except TypeError, ValueError:` | `except (TypeError, ValueError):` |
| 2 | `app/lifecycle.py` | 25 | `except TypeError, ValueError:` | `except (TypeError, ValueError):` |
| 3 | `app/nats/router.py` | 92 | `except asyncio.CancelledError, asyncio.TimeoutError:` | `except (asyncio.CancelledError, asyncio.TimeoutError):` |
| 4 | `app/jobs/record_usages.py` | 497 | `except ValueError, TypeError:` | `except (ValueError, TypeError):` |
| 5 | `app/db/crud/admin.py` | 70 | `except DetachedInstanceError, InvalidRequestError:` | `except (DetachedInstanceError, InvalidRequestError):` |
| 6 | `app/db/crud/general.py` | 217 | `except ValueError, AttributeError:` | `except (ValueError, AttributeError):` |
| 7 | `app/core/xray.py` | 233 | `except IndexError, TypeError:` | `except (IndexError, TypeError):` |
| 8 | `app/core/hosts.py` | 464 | `except ValueError, TypeError:` | `except (ValueError, TypeError):` |

### Example of original code:

## Finding 2: HIGH — Missing Database Indexes (No Schema Migration Needed)

**Severity: High**
**Impact:** Frequently queried columns lack standalone indexes, causing full table scans on large datasets.

### Indexes to add via CRUD initialization or startup hook (using `CREATE INDEX IF NOT EXISTS`):

| Table | Column(s) | Index Name | Rationale |
|-------|-----------|------------|-----------|
| `users` | `status` | `ix_users_status` | Queried in `expire_users_job`, `limit_users_job`, `on_hold_to_active_users_job` |
| `users` | `admin_id` | `ix_users_admin_id` | FK column, no standalone index (only composite) |
| `nodes` | `status` | `ix_nodes_status` | Filtered in `node_health_check` via `NodeListQuery(status=...)` |
| `notification_reminders` | `user_id` | `ix_notification_reminders_user_id` | FK column with no index |
| `hosts` | `inbound_tag` | `ix_hosts_inbound_tag` | FK column, frequent joins |
| `hosts` | `is_disabled` | `ix_hosts_is_disabled` | Frequently filtered |
| `temp_keys` | `action` | `ix_temp_keys_action` | Queried by action |

### Implementation approach:
Add a new function `create_indexes_if_not_exists()` in `app/db/__init__.py` or `app/lifecycle.py` that runs during startup using raw SQL `CREATE INDEX IF NOT EXISTS` — no Alembic migration required.

## Finding 3: HIGH — Untracked asyncio.create_task() Fire-and-Forget Tasks

**Severity: High**
**Impact:** Tasks created with `asyncio.create_task()` without storing references can be silently cancelled during event loop shutdown, causing lost notifications. Python 3.11+ emits `PendingDeprecationWarning` for this.

### Locations:

| File | Line | Code |
|------|------|------|
| `app/jobs/review_users.py` | 46, 50 | `asyncio.create_task(notification.user_data_reset_by_next(...))` |
| `app/jobs/remove_expired_users.py` | 20 | `asyncio.create_task(notification.remove_user(...))` |
| `app/jobs/reset_user_data_usage.py` | 31, 34 | `asyncio.create_task(notification.reset_user_data_usage(...))` |
| `app/jobs/reset_node_usage.py` | 38 | `asyncio.create_task(notification.reset_node_usage(...))` |
| `app/node/sync.py` | 121, 126, 134, 142 | `asyncio.create_task(_dispatch_user_update(...))` |
| `app/routers/admin.py` | 52, 57, 61, 78 | `asyncio.create_task(notification.admin_login(...))` |

### Fix approach:
Create a module-level `_background_tasks: set[asyncio.Task]` set in each file (or a shared utility) and use a helper to track tasks:

```python
_background_tasks: set[asyncio.Task] = set()

def create_background_task(coro) -> asyncio.Task:
    task = asyncio.create_task(coro)
    _background_tasks.add(task)
    task.add_done_callback(_background_tasks.discard)
    return task
```

Replace all bare `asyncio.create_task(...)` calls with `create_background_task(...)`.

## Finding 4: MEDIUM — Timezone-Aware vs Naive Datetime Comparison

**Severity: Medium**
**Impact:** In `app/jobs/send_notifications.py:125-131`, `NotificationReminder.expires_at` is `DateTime(timezone=True)` (timezone-aware), but the comparison uses a naive datetime (`now_naive`). While SQLAlchemy pushes this to SQL and most databases handle it, this is fragile and can fail on certain driver configurations.

### Original code:
```python
async def delete_expired_reminders() -> None:
    async with GetDB() as db:
        now_utc = dt.now(tz=tz.utc)
        now_naive = now_utc.replace(tzinfo=None)
        result = await db.execute(
            delete(NotificationReminder).where(
                NotificationReminder.expires_at < now_naive
            )
        )
```

### Fix:
```python
async def delete_expired_reminders() -> None:
    async with GetDB() as db:
        now_utc = dt.now(tz=tz.utc)
        result = await db.execute(
            delete(NotificationReminder).where(
                NotificationReminder.expires_at < now_utc
            )
        )
```

## Finding 5: MEDIUM — Subscription Token Has No Expiration

**Severity: Medium**
**Security Impact:** In `app/utils/jwt.py:59-64`, the custom subscription token format `v3,{user_id},{timestamp}` stores a creation timestamp but never validates it against a TTL. Compromised tokens remain valid indefinitely unless the user is deleted or `sub_revoked_at` is set.

### Original code:
```python
async def get_subscription_payload(token: str) -> dict | None:
    # ... decode token ...
    return {
        "user_id": u_user_id,
        "created_at": datetime.fromtimestamp(u_created_at, tz=timezone.utc),
    }
```

### Fix approach:
Add optional TTL validation in `get_subscription_payload()`. Use `subscription_env_settings` or a new setting for the TTL. If the token is older than the TTL AND the user hasn't been online recently, reject it. Keep backward-compatible by making this opt-in (default: no change, existing behavior preserved).

```python
# In get_subscription_payload(), after decoding:
# Optional: validate token age against a configurable max lifetime
# Only reject if the token is very old AND the user has been revoked
# This is backward-compatible: without the setting, behavior is unchanged
```

**Note:** This is advisory — full implementation depends on whether the project wants token expiration. The fix should be opt-in via a new env var `SUBSCRIPTION_TOKEN_MAX_LIFETIME_MINUTES` with a default of `0` (disabled).

## Finding 6: LOW — JWT Secret Key Cached Indefinitely

**Severity: Low**
**Impact:** In `app/utils/jwt.py:14`, `@cached()` on `get_secret_key()` has no TTL. If the JWT secret is rotated in the database, existing worker processes continue using the stale key until restart.

### Fix:
Add a TTL to the cache (e.g., 5 minutes) so secret rotation propagates within a reasonable window:

```python
from aiocache import cached, TTLCache

@cached(ttl=300)  # 5 minutes
async def get_secret_key():
    async with GetDB() as db:
        key = await get_jwt_secret_key(db=db)
        return key
```

## Finding 8: LOW — Inefficient MySQL Cleanup Path

**Severity: Low**
**Impact:** In `app/jobs/cleanup_subscription_updates.py:34-51`, the MySQL path loops through each user individually, executing a SELECT + DELETE per user — O(N*Users) queries. The PostgreSQL/SQLite path handles this more efficiently.

### Fix approach:
For MySQL, batch the cleanup by selecting all excess IDs in a single query using a window function approach, then delete them in one statement.

## Finding 9: MEDIUM — `max_instances=30` Global Scheduler Default

**Severity: Medium**
**Impact:** In `app/scheduler.py:4`, the global `max_instances=30` means up to 30 instances of the same job could run simultaneously if a job forgets to set `max_instances=1`. While individual jobs currently set `max_instances=1`, this is a safety concern.

### Fix:
```python
scheduler = AsyncIOScheduler(
    job_defaults={"max_instances": 1, "coalesce": True},
    timezone="UTC"
)
```

This is safe because all individual jobs already set `max_instances=1` and `coalesce=True`.
